The basics of Idle Game Maker
<b>This page includes help on every aspect of game development within Idle Game Maker.</b> Continue reading or pick a topic in the side-bar!
What is Idle Game Maker?
<b>Idle Game Maker</b> is an online tool that lets you create <b>idle games</b>, simply by <b>writing a text file</b>. No programming knowledge needed!
u:an idle game is a game that revolves around waiting for resources to build up, like Cookie Clicker.
u:while idle games are the main focus of Idle Game Maker, the engine is very flexible and you could make all sorts of games with enough creativity!
A word of warning
<div class="warning" style="overflow:hidden;">Idle Game Maker is one of our less-updated projects and may still contain bugs, quirks, and oversights. Features will be added, changed and fixed regularly, and we cannot guarantee 100% smooth sailing at this point. Please be patient, and don't forget to report bugs (or ideas!) on our <a href="https://discordapp.com/invite/cookie" target="_blank">Discord server</a> (we've got a channel just for Idle Game Maker).<div style="clear:left;float:right;font-style:italic;">-Orteil</div></div>
How do I make a game?
An Idle Game Maker game is simply a text file that our website converts into a browser game.
To make your own game, you will have to write that text file using the commands and syntax described below (you'll see, there's a whole bunch of cool things you can do). Here's what a finished game file looks like : <a href="https://orteil.dashnet.org/igm/games/bunnyclicker.txt" target="_blank">https://orteil.dashnet.org/igm/games/bunnyclicker.txt</a>.
You will then have to find a place online to store your text file (<a href="http://pastebin.com/" target="_blank">pastebin.com</a> is a good one; your text files are hosted for free and, if you register, you can edit them as much as you want. <a href="http://www.topsite.com/best/paste" target="_blank">Here are some other text-hosting sites</a>).
Once you're happy with your new game, all that remains to do is to share it with the world!
How do I play my game?
To play a game, take its source file's URL and append it to the following :
u:<b>https://orteil.dashnet.org/igm/?g=</b>
For instance, a game hosted at :
u:<b>www.example.com/mygame.txt</b>
...would be playable at :
u:<b>https://orteil.dashnet.org/igm/?g=www.example.com/mygame.txt</b>
//
A few things of note :
u:if your game is hosted on pastebin.com, you only have to refer to its identifier (the letters at the end of its url). For example, <a href="http://pastebin.com/9HYzgPWX" target="_blank">http://pastebin.com/9HYzgPWX</a> becomes <a href="https://orteil.dashnet.org/igm/?g=9HYzgPWX" target="_blank">https://orteil.dashnet.org/igm/?g=9HYzgPWX</a>.
u:<b>Note :</b> when hosting your file on pastebin.com, make sure you're registered - or you won't be able to edit your game later! Also make sure to set the file to never expire, and to make it either public or unlisted.
u:if you're not hosted on pastebin.com, make sure your file is a .txt file! Your game won't work with any other file type.
General tips about game-making
Some suggestions on how to make a fun game :
u:You can make an idle game about practically <i>anything</i>. Just look around you!
u:Spend some time examining other games you like. You'll probably learn something from the gameplay or from the source file!
u:Test your game. <i>A lot.</i>
u:If people seem to like your game, consider making a reddit post on <a href="http://www.reddit.com/r/idlegamemaker/" target="_blank">r/idlegamemaker</a> where players can talk about it and discuss new features. Having community feedback is essential when making a good game!
u:Keep a separate version for testing only, so you don't accidentally ruin your players' saves.
u:Keep backups of your game's source - you never know!
u:Add colorful descriptions to the elements of your game and, if possible, pictures! Flavor is everything.
u:Don't mindlessly copy other cool games you've seen. Add something of your own, be creative!
u:Spelling mistakes can quickly make a game appear unprofessional. Use spell-check!
u:Keep your game fresh and your players curious by adding new stuff regularly!
And now for some rules
Before you start making rad new idle games, you might want to read (and follow) these guidelines :
u:<b>All games made using this engine are copyrighted to their respective owners.</b>
u:<b>Content creators are expected to respect copyright laws. No user-created content is hosted on our servers.</b>
u:Don't use <b>illegal content</b> in your game!
u:Please no games involving <b>overly lewd, nude or violent</b> stuff.
u:You are allowed to link <b>any images hosted on our servers</b> (any file whose url begins with "https://dashnet.org/" or "https://orteil.dashnet.org/") in your Idle Game Maker games! (If you want to reuse all our Cookie Clicker assets and make your own better version of Cookie Clicker in our engine, that's up to you!)
u:Do not use <b>copyrighted images</b> (be careful with google image search)! If you use content that demands proper attribution, make sure to provide it somewhere!
u:Don't do anything shifty with our engine! Basically, <b>if it feels wrong, don't do it.</b>
u:Don't try to <b>mislead the players</b> into visiting websites they don't want to.
u:No circumventing our system to include <b>scripts</b> or other disallowed HTML tags in your games!

Feel free to just grab one of the example games and replace things to make it your own!

Formatting
You have a certain degree of freedom when writing source files.
u:Leading tabs and spaces are completely ignored; feel free to use them if it helps make your source code clearer.
u~:You can mark a line as a comment with <code>//</code> :
(<code>//</code> needs to be at the start of the line)
<code>
//this is a comment, it will be ignored by the engine
</code>
~u
u~:You can create whole blocks of comments with <code>/* */</code> :
(<code>/*</code> needs to be at the start of a line, <code>*/</code> at the end of another)
<code>
/*
none of this
will be read
by the engine
*/
</code>
~u
Sections
A game's source file is divided into <b>sections</b>. Each section (save for the first one) can be re-ordered as you like or omitted entirely.
You will find detailed information about each section later on, but here's a quick description first :
u:<code>Let's make a game!</code> - this is the section where you can specify the game's name, its author, and some other information.
u:<code>Settings</code> - this is where you can set general things such as the background image, the stylesheets and spritesheets, or the cost increase ratio.
u:<code>CSS</code> - contains raw CSS to style your game further.
u:<code>Includes</code> - contains your includes, which are bits of source file you can reuse multiple times later on. This can be a rather powerful feature.
u:<code>Layout</code> - specifies the page layout your game uses, which is completely customizable. If no layout is declared, a default layout will be used.
u:<code>Buttons</code> - start declaring buttons (things you can click on).
u:<code>Resources</code> - start declaring resources (things you can earn and spend).
u:<code>Buildings</code> - start declaring buildings (things you can buy that produce resources).
u:<code>Upgrades</code> - start declaring upgrades (things you can buy once that provide passive bonuses).
u:<code>Items</code> - start declaring items (like upgrades, except you can have multiple of the same type, each with its own properties).
u:<code>Achievements</code> - start declaring achievements (things you earn by doing other things).
u:<code>Shinies</code> - start declaring shinies (things that appear and disappear randomly that can be clicked for bonuses).
//
A quick example of how to use sections :
<code>
Let's make a game!
name:My Totally Original Game
Resources
*pie
name:Pie
desc:A delicious pie.
Upgrades
*applePieRecipe
name:Apple pie recipe
desc:Unlocks the ability to make apple pies!
*cherryPieRecipe
name:Cherry pie recipe
desc:Boy, you'll never guess!
</code>
Things
Your game is made of <b>Things</b>! Each resource, button, building, achievement and so on is a Thing.
Each thing has a unique <b>key</b> (sometimes also referred to as <b>id</b>), a simple identifier made of letters and digits (no spaces or special characters). Keys are important as they are used by things to interact with other things.
Once a thing's key is declared, you can add properties to it, such as its name, picture, effects and so on.
For instance, if you wish to create a building with the key <code>brownCow</code>, you would write :
<code>
Buildings
*brownCow
name:Brown cow
desc:A big friendly cow that generates 1 milk per second.
cost:20 coins
on tick:give 1 milk
</code>
//
Note :
u:You cannot use the same key for two different things.
u:You can declare multiple keys for the same thing by separating them with |, such as <code>*brownCow|brownCows</code> which will let you refer to your brown cows using both <code>brownCow</code> and <code>brownCows</code>; this lets you write more naturally without having to worry about plurals.
u:<b>Important :</b> <i>do NOT change a thing's first key once set!</i> This would mess up your players' saves since the engine would treat this thing as an entirely new object! You can change a thing's name at any time, however.
Other syntax tips
u:Pretty much <b>every command on any component is optional</b>, save for the first line which should be <code>Let's make a game!</code>.
u:If you make a mistake in your source code (or use commands in ways we didn't expect), you might get an error message, or the game might freeze completely. Try to undo your last changes and see where it went wrong!
u:Writing code, even in Idle Game Maker, requires some patience and rigor; our code parser is designed to be fairly forgiving, but still - a single typo or syntax error is all it takes for your game to not run correctly!
u:A good way to learn more about our syntax is to check out the source for <a href="https://pastebin.com/raw/9HYzgPWX" target="_blank">Bunny Clicker</a>, a simple game specifically made to showcase many of Idle Game Maker's functions.
Let's make a game!
<b>Let's get started!</b>
This is the first section in a source file, in which you define the essential information for your game.
Remember : all of the following is optional!
c:Let's make a game!
Begins this section, and starts your game proper. Idle Game Maker needs this in order to understand we're making a game!
c:name:Name
Lets you set a name for your game. Hopefully something cool!
c:author:Name
c:by:Name
It's you!
c:desc:Your description here!
A description of your game which will show up on the info screen.
Accepts <a href="#textEffects">text effects</a>.
c:forum post:X
<div class="warning" style="overflow:hidden;"><b>Deprecated:</b> our forums are no longer active and we've since switched to Discord.</div>
This will add a link in the "Info" menu to a DashNet forum post with this post number.
For instance, if your forum post's url is "http://forum.dashnet.org/discussion/1234/not-a-real-post", you would put in <code>forum post:1234</code>.
Use your forum post to give players a place to provide feedback, report bugs, and share their progress on your game!
c:created:DD/MM/YYYY
This is the date, month and year you created the game.
c:updated:DD/MM/YYYY
This is the date, month and year you last updated the game. You don't have to keep this up to date!</div>
c:version:X
<b>X</b> can be any number such as 1 or 2.13. If defined, it is displayed next to your game's name. Increase it when you add new stuff!
Settings
This section contains miscellaneous options for your game, such as the background picture, the behavior of cost inflation, or your game's stylesheets and spritesheets.
Again, each of these commands is optional.
c:Settings
Begins this section.
c:background:url.jpg
Sets your game's background picture to url.jpg.
c:tiling background:url.jpg
Same as above, but the background picture will be tiled instead of scaled to fill the screen.
c:building cost increase:X%
Defines how much every building's costs should increase every time you buy one; a value of 110% means each time you buy a building, it gets 10% more expensive. 100% would mean no price increase; 200% makes the price double.
The default value is <b>115%</b>.
<b>Note :</b> can be overridden in a specific building with the command <code>cost increase:X%</code>.
c:building cost refund:X%
Defines how much of every building's costs are refunded when selling it. 50% means we get half of its current price back.
The default value is <b>50%</b>.
<b>Note :</b> can be overridden in a specific building with the command <code>cost refund:X%</code>.
c:spritesheet:Name, X by Y, url.jpg
This command defines a new spritesheet named "Name", where each tile is X pixels horizontally by Y pixels vertically, with the source at "url.jpg".
While a thing can use a single picture as an icon (<code>icon:url.jpg</code>), it is often more convenient to use a big picture composed of many little icons arranged in a grid, and then just pick one of those as an icon for your thing.
An example would be <code>spritesheet:icons, 48 by 48, url.jpg</code>, which you could then use to set a thing's icon with <code>icon:icons[3,1]</code> (which means : grab the tile in the spritesheet "icons" at position 3 horizontally, 1 vertically). Note : icons start in the top-left, at [0,0].
You can declare as many spritesheets as you want simply by calling this command several times, as long as they have different names.
u:<b>Note :</b> sprites in Idle Game Maker tend to be 48 by 48 pixels. If you want to use a different image size, you may have to use CSS classes and custom styling.
c:stylesheet:url
Adds a CSS stylesheet to your game. This defines the appearance of your game's interface (more information in <a href="#cSS">the next section</a>).
You can define as many stylesheets as you want simply by calling this command several times; their effects will stack.
<b>The stylesheet must be a .txt file,</b> not a .css file (unless it is one of our stylesheets stored locally).</b>
If no stylesheet is defined, the game uses the rather minimalist theme located at <a href="./stuff/basic.css" target="_blank">/stuff/basic.css</a>.
You can use another predefined stylesheet <code>stylesheet:stuff/bigBlue.css</code> for a nice, big, blue theme.
c:no particles
Some actions, such as buying things or clicking buttons, can create particles that display what things were produced or consumed.
Specifying this command prevents such particles from appearing.
c:no bulk particles
When changing the bulk-buy amount (such as with pressing shift and ctrl), a particle flies out announcing the state of the new bulk-buy amount. This command removes this feature.
`+/*c:bloom:X
Applies a bloom filter to your game, making things look glowy and soft. <b>X</b> is a number from 0 to 100 which determines the filter's intensity.
This can be a rather obnoxious and potentially low-performance effect; be warned that most players will play with filters turned off!*/`

Layout
The layout defines which things go where in your game. For instance, you could define a box that contains all the buttons the player can click, and another box to hold all the upgrades and achievements.
The game's appearance can further be customized with <a href="#cSS">CSS</a>.
<b>You do not need to bother with layout if you don't want to.</b> You can simply omit this section entirely to have a <b>simple, default layout</b> that should fit most games.
Example
A simple example layout would be :
<code>
Layout
*resources
contains:Resources
header:These are your resources.
*buildings
contains:Buildings
header:These are things you can build.
tooltip origin:left
*unlockables
contains:Upgrades, Achievements
names:hide
</code>
c:*boxKey
Creates a new box with the specified key, in this case "boxKey".
Boxes are the building blocks of your game's layout. A box can contain things and other boxes.
Box keys follow the same naming restrictions as thing keys : only letters and numbers allowed.
After a box is created with this command, all following commands apply to this box until a new box is created.
c:contains:X
Defines what things go in this box.
u:<code>contains:Buttons</code> will make all things with that type show up in this box. Works with <code>Resources</code>, <code>Buttons</code>, <code>Buildings</code>, <code>Upgrades</code>, <code>Achievements</code>, <code>Items</code>.
u:<code>contains:tag:someTag</code> will place all things with the specified tag in this box.
u:<code>contains:anotherBoxKey</code> will place the specified box inside this one. You can nest as many boxes into other boxes as you want, allowing for more complex layouts.
u~:
<code>contains:BulkDisplay</code> will display a little box that says something like "Buying 1" or "Selling 50", depending on the player's bulk-buy setting.
You'll usually want to place this in your buildings box, like so : <code>contains:BulkDisplay, Buildings</code>.
~u
u~:
<code>contains:Log</code> will show the log, a scrollable text box that you can log text into through <a href="#logThisIsAMessage">this command</a>.
~u
u:Put multiple things in the same box like so : <code>contains:Resources, tag:foodUpgrade, Buttons, someOtherBox, Log</code>.
c:in:boxKey
Adds this box to another box's contents.
c:header:some text
Defines text that shows up at the top of this box.
Accepts <a href="#textEffects">text effects</a>.
c:footer:some text
Defines text that shows up at the bottom of this box.
Accepts <a href="#textEffects">text effects</a>.
c:class:cssClass
Gives the specified CSS classes to the box.
See also <a href="#classCssClass-1">this command</a>.
c:tooltip origin:originName
When the mouse is on a thing in this box, its tooltip will appear in a different position depending on which originName you specify. Valid values are <code>top</code>, <code>right</code>, <code>bottom</code>, and <code>left</code>.
c:tooltip class:cssClass
The tooltip shown for things in this box will have the specified classes.
c:no tooltip
The things inside this box will not trigger a tooltip.
c:names:show
c:names:hide
Defines whether all things inside this box should show or hide their names.
c:icons:show
c:icons:hide
Defines whether all things inside this box should show or hide their icons.
c:costs:show
c:costs:hide
Defines whether all things inside this box should show or hide their costs.
c:ps:show
c:ps:hide
Defines whether all things inside this box should show or hide their production per second.
c:use default
Your game will use a simple, default layout.
The default layout uses this code :
<code>

</code>
Things : general concepts
The following sections all involve Things - creating them and manipulating them.
While some types of Things have features unique to them (buildings can be bought and sold, you can have multiple items), all Things have some features in common.
These are :
c:*itemKey
The first line in a thing's declaration. This tells our engine you're starting a new Thing! The key provided will be the thing's unique identifier; use <code>*itemKey1|itemKey2|itemKey3</code> to create multiple identifiers that all redirect to this thing.
A key can <b>only contain letters and numbers</b>, with no spaces.
The section in which a thing is created (<code>Buildings</code>, <code>Upgrades</code>, <code>Resources</code>...) defines its type.
c:name:Name
This thing will be named Name.
You can specify a plural by using this syntax : <code>name:Normal name|Plural name</code>
c:desc:Your description here!
Defines a description for your thing; it will show up in tooltips when hovering on the thing.
You'll usually want to describe what the thing does (gives X of a resource, unlocks a new building...) and maybe add some colorful flavor text.
Accepts <a href="#textEffects">text effects</a>.
c:text:Some text here!
If a text property is specified, the thing will not display its name; instead, it will display this text.
This is useful because this property accepts <a href="#textEffects">text effects</a>, letting you display complex, dynamic information.
For instance, a resource could use <code>text:You have [?(this=0)|no gold|[this] gold].</code>, which would display as "You have 3 gold." if the resource's amount is 3, or "You have no gold." if it is 0.
Tooltips will still only show the thing's name and description, not its text.
c:icon:url.jpg
c:icon:spritesheetName[X,Y]<br>
Sets the icon for your thing; it will show up next to its name. This can be either a plain image url, or a spritesheet name as declared in the Settings section followed by [X,Y] - with X being the row of the icon (starting at 0 from the left) and Y being the column of the icon (starting at 0 from the top).
<b>Note :</b> you can give multiple icons to a thing by using the syntax <code>icon:icon1 icon2 icon3...</code>; this will overlay all the icons specified, with the first one being on top and the last one at the bottom. This allows you to mix and match icons at will.
c:tag:tagName
c:tags:tagName
Gives tags to the thing. You can declare multiple tags by separating them by spaces (<code>tags:tag1 tag2 tag3...</code>). Tags follow the same naming restrictions as thing keys : only letters and numbers allowed.
Tags do nothing by themselves, but can be used in several ways - a layout box could have <code>contains:tag:magicSpell</code>, and all things with the tag "magicSpell" would be placed into it; an upgrade could have the effect <code>multiply yield of tag:worker by 3</code> which would boost all things with the tag "worker".
c:costs:X
Defines the thing's costs, if it is a building or an upgrade.
X is in the form <code>30 gold, 5 XP, 2 magic</code>.
Costs on buildings are susceptible to scaling with the owned amount.
c:req:X
Defines some requirements for the thing to unlock. This has various effects depending on the thing's type.
If the thing is a resource, button, building or upgrade, it will not appear unless the specified requirements are fulfilled. It will disappear again if it doesn't match the requirements (unless it is an upgrade and you own it).
If the thing is an achievement, it will be earned if the specified requirements are fulfilled.
If the thing is a shiny, it will only spawn if the requirements are fulfilled.
Requirements can be simple amount checks, or full condition <a href="#expressions">expressions</a>.
Some examples :
u:<code>req:3 gems</code> - the thing will only unlock if you have at least 3 gems.
u:<code>req:15 power per second</code> - the thing will only unlock if you're producing 15 of the power resource per second.
u:<code>req:have archeryUpgrade</code> - the thing will only unlock if you own the archery upgrade.
u:<code>req:(waterLevel<20 and chance(50%))</code> - the thing will only unlock if the water level is less than 20, with a 1 in 2 chance.
You can also stack several requirements together, such as :
<code>
req:enemyHP&lt;enemyMaxHP*0.1
req:have finishAbility
</code>
The thing in this example would be a "Finish it!" button that shows up when the enemy is at low health.
According to the above requirements, it will only show up if the enemy's health points are less than 10% of their maximum and you own the "finishAbility" upgrade.
c:hidden
The thing will not be displayed.
c:no text
The thing will not display its name or cost, only its icon (if it has one).
c:class:cssClass
Give some CSS classes to the thing.
Multiple classes can be specified by separating them by spaces : <code>cssClass1 cssClass2 cssClass3...</code>.
Idle Game Maker comes preloaded with a few handy classes, so you can give your main button the class <code>class:bigButton hasFlares</code> to make it show as a large transparent button with a swirling halo behind it.
If you declared a custom stylesheet in Settings or are using additional CSS that adds more classes, you can use those classes here too.
c:icon class:cssClass
Gives CSS classes specifically to the thing's icon element.
c:tooltip origin:originName
When the mouse is on the thing, its tooltip will appear in a different position depending on which originName you specify. Valid values are <code>top</code>, <code>right</code>, <code>bottom</code>, and <code>left</code>.
c:tooltip class:cssClass
The thing's tooltip will have the specified CSS classes.
c:no tooltip
The thing will not trigger a tooltip.
c:start with:X
If the thing is a resource or a building, the player will start with X of it.
c:owned
c:start with
If the thing is an upgrade or an achievement, the player will own them at game start.
c:shown
The thing will be visible when the game starts.
c:hidden
The thing will be hidden when the game starts.
c:lit
The thing will be lit when the game starts. This does nothing except add the CSS class "lit" to the thing.
c:dim
The thing will be unlit when the game starts. This does nothing except add the CSS class "dim" to the thing.
c:hidden when 0
The thing will be hidden if its amount is 0 or less (if it is a resource or building) or if you do not own it (if it is an upgrade or achievement).
c:always hidden
The thing will never be displayed.
Buttons
<b>Buttons</b> are things whose main purpose is to be clicked, usually to produce a resource.
Any thing can behave as a button by giving it <code>on click</code> effects, but buttons (and shinies) are the only elements that keep track of how many times they've been clicked.
Example button
&nbsp;
<code>
*scratchNose
name:Scratch nose
desc:Click this to scratch your nose. Grants 1 satisfaction per click.
on click:yield 1 satisfaction
</code>
c:show clicks
The button's tooltip will display how many times it's been clicked.
c:(thingKey:clicks)
In an expression, returns how many times the button has been clicked.
Resources
<b>Resources</b> are things you can gain and spend.
You can use resources as currencies for purchasing buildings and upgrades, as timers, or as hidden counters for more complicated game mechanics.
Example resource
&nbsp;
<code>
*fuel
name:Fuel
desc:This is fuel for your rocket. You can't visit planets if you don't have enough fuel!
</code>
c:can be negative
Allows the resource's amount to go below 0.
c:is always:X
Sets the resource's amount to always be the specified value.
For instance, if you wanted a resource to track the lowest value ever reached by another resource, you would do <code>is always:min(otherResource)</code>.
c:show earned
The resource's tooltip will display its earned amount.
c:show max
The resource's tooltip will display its max amount.
c:(thingKey)
In an expression, returns the resource's current amount.
c:(thingKey:earned)
In an expression, returns the resource's total earned amount (this goes up with gains, but does not go down with losses).
c:(thingKey:max)
In an expression, returns the resource's highest amount it's ever reached.
c:(thingKey:ps)
In an expression, returns how much the resource is gaining (or losing) per second.
(Note : only gains and losses that happen in an on tick effect block count towards this number.)
Buildings
<b>Buildings</b> are things you can buy and sell, which increases or lowers their amount. Their resource yield and costs are scaled by their amount.
A building can be clicked to purchase one of it; keyboard shortcuts can be used to bulk-buy and sell the building (shift-click for bulk-buying 50, ctrl-click for selling 1, ctrl-shift-click for bulk-selling 50).
A building's effects only kick in when you have at least 1 of it.
Example building
&nbsp;
<code>
*cheesecakeFactory
name:Cheesecake factory
desc:Combines the magic of cheese with the wonders of cake to produce 3 cheesecakes per second.
on tick:yield 3 cheesecakes
cost:100 money
</code>
c:cost increase:X%
Defines how much the building's costs should increase every time you buy one; a value of 110% means each time you buy 1, it gets 10% more expensive. 100% would mean no price increase; 200% makes the price double.
The default value is 115%.
Also see <code>building cost increase:X%</code> in the Settings section.
c:cost refund:X%
Defines how much of the building's costs are refunded when selling it. 50% means we get half of its current price back.
The default value is 50%.
Also see <code>building cost refund:X%</code> in the Settings section.
c:no buy
The building cannot be bought or sold by clicking it. Use this if you'd like to handle changing the building's amount using your own mechanics.
c:show max
The building's tooltip will display its max amount.
c:(thingKey)
In an expression, returns the building's current amount.
c:(thingKey:max)
In an expression, returns the building's highest amount it's ever reached.
Upgrades
<b>Upgrades</b> are things you can purchase. They usually confer passive boosts.
An upgrade can be clicked to purchase it.
An upgrade's effects only kick in when you own it.
Example upgrade
&nbsp;
<code>
*kungfuTraining
name:Kung-fu training
desc:Everybody's doing it!
passive:multiply yield of attackButton by 2
cost:3 trainingPoints
</code>
c:no buy
The upgrade cannot be bought by clicking it. Use this if you'd like to handle getting the upgrade using your own mechanics.
c:(thingKey)
In an expression, returns whether or not we own the upgrade.
Achievements
<b>Achievements</b> are things you can unlock, usually as recognition for some milestone.
An achievement may have passive effects just like an upgrade, in which case the effects only kick in when you own it.
Example achievement
&nbsp;
<code>
*10clicks
name:Wowie! Ten whole clicks!
desc:You've clicked the big red button 10 times. Well done. Well done indeed.
req:10 bigButton:clicks
</code>
c:(thingKey)
In an expression, returns whether or not we own the achievement.
Items
<b>Items</b> are things you can own multiple instances of; each instance may have its own properties.
A player can only own up to <b>100</b> item instances. Attempting to give them more will fail. You can check how many items you can still generate through the variable <code>ItemsLeft</code>, which logically ranges from 100 (no items) to 0 (all item slots used up).
Example item
&nbsp;
<code>
*sword
name:A sword
desc:A very generic sword. +3 attack.
passive:grant 3 attack
</code>
Shinies
<b>Shinies</b> are things that appear randomly on the screen, linger around for a bit, then disappear again.
Shinies, like buttons, keep track of how many times they've been clicked.
Example shiny
&nbsp;
<code>
*luckyGnome
name:Lucky gnome
on click:yield (random(10,100)) gold
movement:anywhere fade bounce moveRandom
frequency:120
</code>
c:frequency:X
The shiny will spawn every X seconds.
<b>The shiny will not spawn by itself unless you define a frequency.</b> You will, however, be able to manually spawn it through effects.
c:frequency variation:X
The shiny may take up to an additional X seconds to spawn.
A shiny with <code>frequency:30</code> and <code>frequency variation:20</code> may take between 30 and 50 seconds to spawn.
c:duration:X
The shiny will stay on screen for X seconds before it disappears. Defaults to 10 seconds.
c:no click
The shiny will not respond to clicks; it will be purely visual.
You could use this creatively to employ some shinies as animated visual elements.
c:movement:X
Defines the behavior the shiny displays while on screen.
You can mix and match several behaviors like so : <code>movement:grow anywhere randomAngle fade</code>.
u:<code>anywhere</code> : the shiny will spawn on a random spot on the screen
u:<code>onLeft</code>, <code>onRight</code>, <code>onTop</code>, <code>onBottom</code> : the shiny will spawn on the specified edge of the screen
u:<code>onMouse</code> : the shiny will spawn on the mouse
u:<code>onThing:thingKey</code> : the shiny will spawn on the specified thing if that thing is visible
u:<code>onBox:boxKey</code> : the shiny will spawn on the specified box if that box is visible
u:<code>followMouse</code> : the shiny will stick to the mouse as it moves around
u:<code>followMouseSlow</code> : the shiny will stick to the mouse, but with some slack (you can define how much slack with <code>followMouseSlow:X</code>)
u:<code>wiggle</code> : the shiny will rotate softly back and forth (set a custom speed with <code>wiggle:X</code>)
u:<code>pulse</code> : the shiny will pulse (set a custom speed with <code>pulse:X</code>)
u:<code>randomAngle</code> : the shiny will spawn rotated in a random direction
u:<code>spinCW</code> : the shiny will spin clockwise (set a custom speed with <code>spinCW:X</code>)
u:<code>spinCCW</code> : the shiny will spin counter-clockwise (set a custom speed with <code>spinCCW:X</code>)
u:<code>spinRandom</code> : the shiny will spin clockwise or counter-clockwise (set a custom speed with <code>spinRandom:X</code>)
u:<code>fade</code> : the shiny will fade in and out over its lifetime
u:<code>grow</code> : the shiny will grow over its lifetime
u:<code>shrink</code> : the shiny will shrink over its lifetime
u:<code>growShrink</code> : the shiny will grow and then shrink over its lifetime
u:<code>moveRandom</code> : the shiny will move towards a random direction (set a custom speed with <code>moveRandom:X</code>)
u:<code>moveLeft</code>, <code>moveRight</code>, <code>moveTop</code>, <code>moveBottom</code> : the shiny will move towards the specified edge of the screen over its lifetime
u:<code>bobVertical</code> : the shiny will bob up and down (set a custom speed with <code>bobVertical:X</code>)
u:<code>bobHorizontal</code> : the shiny will bob left and right (set a custom speed with <code>bobHorizontal:X</code>)
u:<code>bounce</code> : the shiny will bounce up and down (set a custom speed with <code>bounce:X</code>)
c:(thingKey:clicks)
In an expression, returns how many times the shiny has been clicked.
Text effects
These are effects you can apply to text in descriptions, logs and others.
Embeds
You can embed expressions in text.
For instance :
u:<code>[thingKey]</code> will be replaced by the amount of the given thing, if it is a resource or a builing.
u:<code>[thingKey:max]</code>, <code>[thingKey:ps]</code>, <code>[thingKey:clicks]</code> and others are all valid too.
u:<code>[3*thingKey+19]</code> or any other mathematical expression will display the expression's result.
u:<code>[s?thingKey]</code> will be replaced by a "s" if the specified thing's amount is different than 1. For instance : <code>You have [gems] gem[s?gems].</code>
u:<code>[n:thingKey]</code> will give the name of the specified thing. Use <code>[n:this]</code> to refer to the current thing.</code>
u:<code>[p:thingKey]</code> will give the plural of the specified thing. Use <code>[p:this]</code> to refer to the current thing.</code>
u:You can add text conditionally with <code>[?thingKey or expression|text to replace if true|text to replace if false]</code>; for instance, <code>You [?(have kungFu)|do|do not] know the secrets of kung fu.</code> will be replaced by "You do know the secrets of kung fu" if you own the kungFu upgrade, or "You do not know the secrets of kung fu" if not.
Tags
You can use <b>tags</b> in text to make it nicer-looking.
List of tags :
u:<code>&lt;b&gt;text&lt;/b&gt;</code> will produce <b>text</b>
u:<code>&lt;i&gt;text&lt;/i&gt;</code> will produce <i>text</i>
u:<code>&lt;u&gt;text&lt;/u&gt;</code> will produce <u>text</u>
u:<code>&lt;t&gt;text&lt;/t&gt;</code> will style the text as a big title
u:<code>&lt;q&gt;text&lt;/q&gt;</code> will style the text as a quote (use this for flavor text in descriptions)
u:<code>&lt;#f00&gt;text&lt;/#&gt;</code> will display the text in the given hex color
u:<code>&lt;/&gt;</code> will produce a linebreak
u:<code>&lt;//&gt;</code> will start a new paragraph
u:<code>&lt;.&gt;something or other</code> will produce a dotted list item (use this to list effects in descriptions)
<b>Warning :</b> if a tag requires a closing tag (like <code>&lt;b&gt;this&lt;/b&gt;</code>), omitting the closing tag may break the game's layout!
Effects
Effects are a powerful feature that lets any thing do just about anything.
Effects are used to produce resources, generate items, chance prices, boost production, and do different things based on conditions.
You can declare an effect in one line :
<code>
on click:yield 1 chocolate
</code>
...or on multiple lines, if your effects are more complex (this is called an effect block) :
<code>
on click:
yield 1 chocolate
if (have magicChocolateUpgrade) yield 3 cocoaSparkles
if (chance(0.1%) yield rareCocoaPod
end
</code>
<b>Note the final <code>end</code>!</b> This is easy to forget, but multiline effects should always end with it.
You can define multiple of the same effect, in which case they will all stack sequentially :
<code>
passive:yield 3 beans
passive:multiply yield of beans by 1.1
</code>
...is the same as :
<code>
passive:
yield 3 beans
multiply yield of beans by 1.1
end
</code>
Types of effects
There are a few built-in effect types which are triggered at different times.
u~:
<code>on start:</code> will trigger once when a new game is launched, such as when the player first opens the game or wipes their save.
~u
u~:
<code>on save:</code> will trigger right before the game saves or is exported.
~u
u~:
<code>on load:</code> will trigger after the game is loaded, be it from opening the game, importing a save, or manually reloading.
~u
u~:
<code>on tick:</code> will trigger every <b>game tick</b> - that is, every second; use this for resource production by buildings.
If the thing is a building, this will only fire if its amount is at least 0.
If the thing is an upgrade or an achievement, this will only fire if you own it.
<b>Example :</b>
<code>
on tick:yield 1 apple
</code>
~u
u~:
<code>on click:</code> will trigger when the thing is clicked; use this for button and shiny effects.
Many effects, such as passive boosts, do not work in on click events, and must be placed in passive effect blocks instead.
<b>Example :</b>
<code>
on click:yield 1 apple
</code>
~u
u~:
<code>passive:</code> will trigger internally to grant various boosts; use this for upgrade effects.
<b>Example :</b>
<code>
passive:grant 10 maxScience
passive:multiply yield of makeScienceButton by 1.5
passive:multiply cost of scienceMachine by 0.9
</code>
(Tip : passive effect blocks and on tick effect blocks are actually handled as the same block by the engine but are differentiated for clarity.)
~u
u:<code>on earn:</code> will trigger when the thing's amount is increased or (if it is an upgrade or achievement) if it is earned.
u:<code>on lose:</code> will trigger when the thing's amount is decreased or (if it is an upgrade or achievement) if it is lost.
Custom effects
You may also declare custom effects, which can be called from other effects (see <a href="#doEffectWithThingSelector">do effect with thingSelector</a>).
For instance :
<code>
on explore:
log You explore [n:this]!
gain 1 landsExplored
end
</code>
...which you would trigger with :
<code>
on click:
do explore with jungleLand
end
</code>
Effect names follow the same naming restrictions as thing keys : only letters and numbers allowed.
Condition flow
Effects may include <b>conditions</b>, which are <a href="#expressions">expressions</a> that follow if/else logic to determine if the subsequent effects are applied or not.
For instance :
<code>
on click:if (someResource>=50) yield someUpgrade
</code>
Condition expressions should always be in parentheses.
Multiline condition blocks should finish with <code>end</code>, like multiline effects :
<code>
on click:
if (someResource>=50)
yield someUpgrade

Effects may use <b>local variables</b>. A local variable is a value you can set, change, and use within an effect block. Once the effect block is finished, all local variables are reset.
A variable's name starts with $ and contains only letters or digits.
Variables are often used for convenience when doing complex operations.
An example :

Expressions
Some commands, such as effects and unlock requirements, can accept <b>expressions</b>. Expressions allow you to use arbitrary mathematic operations to produce values or condition checks.
For instance, say you want to multiply the player's gold resource by 3; you would have an effect that does <code>set gold to gold*3</code>. You could also have a button that gives between 0 and 5 gems on click - this would be <code>yield random(0,5) gems</code>. An achievement that only unlocks when the player has clicked an "attack" button 50 times or more would have the command <code>req:attackButton:clicks>=50</code>.
An expression that contains spaces (such as <code>(have thisUpgrade and thatResource>10)</code>) should be wrapped in parentheses.
Values
Expressions transform values to produce a result.
Values can be simple numbers, such as <code>41</code> or <code>2.7</code>; they can also be things referenced by their key - <code>myBuilding</code> would represent how many of that building the player has.
There's many ways to get creative with values; for instance, <code>random(3,enemyLevel)</code> would give a random number between 3, and the amount of the "enemyLevel" resource.

Effects and unlock requirements may involve conditions, which are simply expressions that are considered false if they equal exactly 0, or true otherwise.
For instance :
<code>on click:if (have canClick) yield 1 coolPoint</code>

Templates
Templates let you declare a pseudo-thing with its own commands; all new things declared after it will copy its commands.

When a thing declares its own command, it overrides whatever value the template previously set, except for certain commands (such as effects) where the new values are simply added to what the template already set.
Templates are cleared when you declare a new template or start a new section.

Includes
Includes are an advanced feature that lets you create complex source files while avoiding code repetition.
You use includes by <b>first defining an include</b> in the Includes section, then <b>calling it as many times as you want</b> later in the code. Each include call is replaced by its full include text before being executed as source code.
Structure
Let's take a simple example include and cut it up :
<code>*include multiplyBy3 %value : %value*3</code>
u:<code>*include</code> - starts defining a new include.
u:<code>multiplyBy3</code> - the next word after <code>*include</code> sets the include's name.
u~:
<code>%value</code> - adds a parameter.
An include can have multiple parameters, such as <code>*include sayThing %name %textToSay</code>.
A parameter can have a default value which will be used if an include call doesn't specify that parameter; this is done like so : <code>*include sayThing %name="Anonymous" %textToSay="..."</code>.
~u
u:<code> : </code> - defines this include as a single-line include, meaning we can then call it from inside other lines (see the following Examples section for an example of a multi-line include). Everything after the <code> : </code> is the include's content.
u:<code>%value*3</code> - the include's content, which any call to this include will be replaced by. This can be pretty much anything you want, and usually involves the parameters the include defined.
//
Let's now look at a call for that same include :
<code>on click:gain [i:multiplyBy3 %value="20"] gold</code>
u:<code>[i:</code> - signals that we are calling an include.
u:<code>multiplyBy3</code> - this is the name of the include we're calling.
u:<code>%value="20"</code> - sets the parameter "%value" to "20". This means everytime "%value" is encountered in the include's content, it will be replaced here with "20".
u:<code>]</code> - closes the include call.
Examples
A very simple, one-line, pretty much useless include :
<code>
Includes
*include multiplyBy3 %value : %value*3
</code>
Consequently, if you were to write :
<code>[i:multiplyBy3 %value="4"]</code>
...it would automatically be replaced in your source code by :
<code>4*3</code>
You could then use it like so :
<code>
log 11 times 3 is [[i:multiplyBy3 %value="11"]].
on click:gain [i:multiplyBy3 %value="20"] gold
</code>
//
The following code creates a multi-line include that lets you create entire upgrades in one include call.
<code>
Includes
*include food %id %name %flavor="alright"
*food[%id]
name:[%name]
desc:This [%name] is [%flavor].
</code>
You would use it like so :
<code>
Upgrades
